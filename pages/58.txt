<p>In <a href="/wiki/Computing" title="Computing">computing</a>, an <b>interpreter</b> is <a href="/wiki/Software" title="Software">software</a> that <a href="/wiki/Execution_(computers)" class="mw-redirect" title="Execution (computers)">executes</a> <a href="/wiki/Source_code" title="Source code">source code</a> without first <a href="/wiki/Compiling" class="mw-redirect" title="Compiling">compiling</a> it to <a href="/wiki/Machine_code" title="Machine code">machine code</a>. An interpreted <a href="/wiki/Runtime_environment" class="mw-redirect" title="Runtime environment">runtime environment</a> differs from one that processes <a href="/wiki/CPU" class="mw-redirect" title="CPU">CPU</a>-native <a href="/wiki/Executable_code" class="mw-redirect" title="Executable code">executable code</a> which requires translating source code before executing it. An interpreter may translate the source code to an intermediate format, such as <a href="/wiki/Bytecode" title="Bytecode">bytecode</a>. A hybrid environment may translate the bytecode to machine code via <a href="/wiki/Just-in-time_compilation" title="Just-in-time compilation">just-in-time compilation</a>, as in the case of <a href="/wiki/.NET" title=".NET">.NET</a> and <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, instead of interpreting the bytecode directly.
</p><p>Before the widespread adoption of interpreters, the execution of <a href="/wiki/Computer_programs" class="mw-redirect" title="Computer programs">computer programs</a> often relied on <a href="/wiki/Compilers" class="mw-redirect" title="Compilers">compilers</a>, which translate and compile source code into machine code. Early runtime environments for <a href="/wiki/Lisp_programming_language" class="mw-redirect" title="Lisp programming language">Lisp</a> and <a href="/wiki/BASIC_interpreter" title="BASIC interpreter">BASIC</a> could parse source code directly. Thereafter, runtime environments  were developed for languages (such as <a href="/wiki/Perl" title="Perl">Perl</a>, <a href="/wiki/Raku_(programming_language)" title="Raku (programming language)">Raku</a>, <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, <a href="/wiki/MATLAB" title="MATLAB">MATLAB</a>, and <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>), which translated source code into an intermediate format before executing to enhance <a href="/wiki/Runtime_performance" class="mw-redirect" title="Runtime performance">runtime performance</a>.
</p><p>Code that runs in an interpreter can be run on any platform that has a <a href="/wiki/Software_compatibility" class="mw-redirect" title="Software compatibility">compatible</a> interpreter. The same code can be distributed to any such platform, instead of an <a href="/wiki/Executable" title="Executable">executable</a> having to be built for each platform. Although each programming language is usually associated with a particular runtime environment, a language can be used in different environments. Interpreters have been constructed for languages traditionally associated with <a href="/wiki/Compiler" title="Compiler">compilation</a>, such as <a href="/wiki/ALGOL" title="ALGOL">ALGOL</a>, <a href="/wiki/Fortran" title="Fortran">Fortran</a>, <a href="/wiki/COBOL" title="COBOL">COBOL</a>, <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> and <a href="/wiki/C%2B%2B" title="C++">C++</a>.
</p>

<div class="mw-heading mw-heading2"><h2 id="History">History</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Interpreter_(computing)&amp;action=edit&amp;section=1" title="Edit section: History"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In the early days of computing, compilers were more commonly found and used than interpreters because hardware at that time could not support both the interpreter and interpreted code and the typical batch environment of the time limited the advantages of interpretation.<sup id="cite&#95;ref-1" class="reference"><a href="#cite_note-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Interpreters were used as early as 1952 to ease programming within the limitations of computers at the time (e.g. a shortage of program storage space, or no native support for floating point numbers). Interpreters were also used to translate between low-level machine languages, allowing code to be written for machines that were still under construction and tested on computers that already existed.<sup id="cite&#95;ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> The first interpreted high-level language was <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a>. Lisp was first implemented by <a href="/wiki/Steve_Russell_(computer_scientist)" title="Steve Russell (computer scientist)">Steve Russell</a> on an <a href="/wiki/IBM_704" title="IBM 704">IBM 704</a> computer. Russell had read <a href="/wiki/John_McCarthy_(computer_scientist)" title="John McCarthy (computer scientist)">John McCarthy</a>'s paper, "Recursive Functions of Symbolic Expressions and Their Computation by Machine, Part I", and realized (to McCarthy's surprise) that the Lisp <i>eval</i> function could be implemented in machine code.<sup id="cite&#95;ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> The result was a working Lisp interpreter which could be used to run Lisp programs, or more properly, "evaluate Lisp expressions".
</p><p>The development of editing interpreters was influenced by the need for interactive computing. In the 1960s, the introduction of time-sharing systems allowed multiple users to access a computer simultaneously, and editing interpreters became essential for managing and modifying code in real-time. The first editing interpreters were likely developed for mainframe computers, where they were used to create and modify programs on the fly. One of the earliest examples of an editing interpreter is the EDT (Editor and Debugger for the TECO) system, which was developed in the late 1960s for the PDP-1 computer. EDT allowed users to edit and debug programs using a combination of commands and macros, paving the way for modern text editors and interactive development environments.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (October 2024)">citation needed</span></a></i>&#93;</sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Use">Use</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Interpreter_(computing)&amp;action=edit&amp;section=2" title="Edit section: Use"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Notable uses for interpreters include:
</p>
<dl><dt>Commands and scripts</dt>
<dd>Interpreters are frequently used to execute <a href="/wiki/Command-line_interface" title="Command-line interface">commands</a> and <a href="/wiki/Script_language" class="mw-redirect" title="Script language">scripts</a></dd></dl>
<dl><dt><a href="/wiki/Virtualization" title="Virtualization">Virtualization</a></dt>
<dd>An interpreter acts as a <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machine</a> to execute machine code for a hardware architecture different from the one running the interpreter.</dd></dl>
<dl><dt>Emulation</dt>
<dd>An interpreter (virtual machine) can <a href="/wiki/Emulator" title="Emulator">emulate</a> another computer system in order to run code written for that system.</dd></dl>
<dl><dt><a href="/wiki/Sandbox_(computer_security)" title="Sandbox (computer security)">Sandboxing</a></dt>
<dd>While some types of sandboxes rely on operating system protections, an interpreter (virtual machine) can offer additional control such as blocking code that violates <a href="/wiki/Computer_security" title="Computer security">security</a> rules.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>&#93;</sup></dd></dl>

<dd><a href="/wiki/Self-modifying_code" title="Self-modifying code">Self-modifying code</a> can be implemented in an interpreted language. This relates to the origins of interpretation in Lisp and <a href="/wiki/Artificial_intelligence" title="Artificial intelligence">artificial intelligence</a> research.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. <div class="mw-heading mw-heading2"><h2 id="Efficiency">Efficiency</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Interpreter_(computing)&amp;action=edit&amp;section=3" title="Edit section: Efficiency"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Interpretive overhead is the runtime cost of executing code via an interpreter instead of as native (compiled) code. Interpreting is slower because the interpreter executes multiple machine-code instructions for the equivalent functionality in the native code. In particular, access to variables is slower in an interpreter because the mapping of identifiers to storage locations must be done repeatedly at run-time rather than at <a href="/wiki/Compile_time" title="Compile time">compile time</a>.<sup id="cite&#95;ref-FOLDOC&#95;4-0" class="reference"><a href="#cite_note-FOLDOC-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> But faster development (due to factors such as shorter edit-build-run cycle) can outweigh the value of faster execution speed—especially when prototyping and testing when the edit-build-run cycle is frequent.<sup id="cite&#95;ref-FOLDOC&#95;4-1" class="reference"><a href="#cite_note-FOLDOC-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup>
</p><p>An interpreter may generate an <a href="/wiki/Intermediate_representation" title="Intermediate representation">intermediate representation</a> (IR) of the program from source code in order to achieve goals such as fast runtime performance. A compiler may also generate an IR, but the compiler generates machine code for later execution whereas the interpreter prepares to execute the program. These differing goals lead to differing IR design. Many <a href="/wiki/BASIC" title="BASIC">BASIC</a> interpreters replace <a href="/wiki/Keyword_(computer_programming)" class="mw-redirect" title="Keyword (computer programming)">keywords</a> with single <a href="/wiki/Byte" title="Byte">byte</a> <a href="/wiki/Token_threading" class="mw-redirect" title="Token threading">tokens</a> which can be used to find the instruction in a <a href="/wiki/Jump_table" class="mw-redirect" title="Jump table">jump table</a>.<sup id="cite&#95;ref-FOLDOC&#95;4-2" class="reference"><a href="#cite_note-FOLDOC-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> A few interpreters, such as the <a href="/wiki/PBASIC" title="PBASIC">PBASIC</a> interpreter, achieve even higher levels of program compaction by using a bit-oriented rather than a byte-oriented program memory structure, where commands tokens occupy perhaps 5 bits, nominally "16-bit" constants are stored in a <a href="/wiki/Variable-length_code" class="mw-redirect" title="Variable-length code">variable-length code</a> requiring 3, 6, 10, or 18 bits, and address operands include a "bit offset". Many BASIC interpreters can store and read back their own tokenized internal representation.
</p><p>There are various compromises between the development speed when using an interpreter and the execution speed when using a compiler. Some systems (such as some <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisps</a>) allow interpreted and compiled code to call each other and to share variables. This means that once a routine has been tested and debugged under the interpreter it can be compiled and thus benefit from faster execution while other routines are being developed.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>&#93;</sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Implementation">Implementation</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Interpreter_(computing)&amp;action=edit&amp;section=4" title="Edit section: Implementation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>

<div class="mw-heading mw-heading2"><h2 id="Just-in-time_compilation">Just-in-time compilation</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Interpreter_(computing)&amp;action=edit&amp;section=6" title="Edit section: Just-in-time compilation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><a href="/wiki/Just-in-time_compilation" title="Just-in-time compilation">Just-in-time (JIT) compilation</a> is the process of converting an intermediate format (i.e. bytecode) to native code at runtime. As this results in native code execution, it is a method of avoiding the runtime cost of using an interpreter while maintaining some of the benefits that led to the development of interpreters.
</p>
<div class="mw-heading mw-heading2"><h2 id="Variations">Variations</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Interpreter_(computing)&amp;action=edit&amp;section=7" title="Edit section: Variations"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<dl><dt><a href="/wiki/Control_table" title="Control table">Control table</a> interpreter</dt>
<dd>Logic is specified as data formatted as a table.</dd></dl>
<dl><dt><span class="anchor" id="Compreter"></span>Bytecode interpreter</dt>
<dd>Some interpreters process <a href="/wiki/Bytecode" title="Bytecode">bytecode</a> which is an intermediate format of logic compiled from a high-level language. For example, <a href="/wiki/Emacs_Lisp" title="Emacs Lisp">Emacs Lisp</a> is compiled to bytecode which is interpreted by an interpreter. One might say that this compiled code is machine code for a virtual machine – implemented by the interpreter. Such an interpreter is sometimes called a <i>compreter</i>.<sup id="cite&#95;ref-Kühnel&#95;1987&#95;Kleincomputer&#95;6-0" class="reference"><a href="#cite_note-Kühnel_1987_Kleincomputer-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-Heyne&#95;1984&#95;Compreter&#95;7-0" class="reference"><a href="#cite_note-Heyne_1984_Compreter-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup></dd></dl>
<dl><dt>Threaded code interpreter</dt>
<dd>A <a href="/wiki/Threaded_code" title="Threaded code">threaded code</a> interpreter is similar to bytecode interpreter but instead of bytes, uses pointers. Each instruction is a word that points to a function or an instruction sequence, possibly followed by a parameter. The threaded code interpreter either loops fetching instructions and calling the functions they point to, or fetches the first instruction and jumps to it, and every instruction sequence ends with a fetch and jump to the next instruction. One example of threaded code is the <a href="/wiki/Forth_(programming_language)" title="Forth (programming language)">Forth</a> code used in <a href="/wiki/Open_Firmware" title="Open Firmware">Open Firmware</a> systems. The source language is compiled into "F code" (a bytecode), which is then interpreted by a <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machine</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>&#93;</sup></dd></dl>
<dl><dt>Abstract syntax tree interpreter</dt>
<dd>An abstract syntax tree interpreter transforms source code into an <a href="/wiki/Abstract_syntax_tree" title="Abstract syntax tree">abstract syntax tree</a> (AST), then interprets it directly, or uses it to generate native code via JIT compilation.<sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup> In this approach, each sentence needs to be parsed just once. As an advantage over bytecode, AST keeps the global program structure and relations between statements (which is lost in a bytecode representation), and when compressed provides a more compact representation.<sup id="cite&#95;ref-KistlerFranz1999&#95;9-0" class="reference"><a href="#cite_note-KistlerFranz1999-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup> Thus, using AST has been proposed as a better intermediate format than bytecode. However, for interpreters, AST results in more overhead than a bytecode interpreter, because of nodes related to syntax performing no useful work, of a less sequential representation (requiring traversal of more pointers) and of overhead visiting the tree.<sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup></dd></dl>
<dl><dt>Template interpreter</dt>
<dd>Rather than implement the execution of code by virtue of a large switch statement containing every possible bytecode, while operating on a software stack or a tree walk, a template interpreter maintains a large array of bytecode (or any efficient intermediate representation) mapped directly to corresponding native machine instructions that can be executed on the host hardware as key value pairs (or in more efficient designs, direct addresses to the native instructions),<sup id="cite&#95;ref-auto&#95;11-0" class="reference"><a href="#cite_note-auto-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-12" class="reference"><a href="#cite_note-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup> known as a "Template". When the particular code segment is executed the interpreter simply loads or jumps to the opcode mapping in the template and directly runs it on the hardware.<sup id="cite&#95;ref-13" class="reference"><a href="#cite_note-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-14" class="reference"><a href="#cite_note-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup> Due to its design, the template interpreter very strongly resembles a JIT compiler rather than a traditional interpreter, however it is technically not a JIT due to the fact that it merely translates code from the language into native calls one opcode at a time rather than creating optimized sequences of CPU executable instructions from the entire code segment. Due to the interpreter's simple design of simply passing calls directly to the hardware rather than implementing them directly, it is much faster than every other type, even bytecode interpreters, and to an extent less prone to bugs, but as a tradeoff is more difficult to maintain due to the interpreter having to support translation to multiple different architectures instead of a platform independent virtual machine/stack. To date, the only template interpreter implementations of widely known languages to exist are the interpreter within Java's official reference implementation, the Sun HotSpot Java Virtual Machine,<sup id="cite&#95;ref-auto&#95;11-1" class="reference"><a href="#cite_note-auto-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup> and the Ignition Interpreter in the Google <a href="/wiki/V8_(JavaScript_engine)" title="V8 (JavaScript engine)">V8</a> JavaScript execution engine.</dd></dl>
<dl><dt>Microcode</dt>
<dd><a href="/wiki/Microcode" title="Microcode">Microcode</a> provides an abstraction layer as a hardware interpreter that implements machine code in a lower-level machine code.<sup id="cite&#95;ref-Kent2813&#95;15-0" class="reference"><a href="#cite_note-Kent2813-15"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup> It separates the high-level machine instructions from the underlying <a href="/wiki/Electronics" title="Electronics">electronics</a> so that the high-level instructions can be designed and altered more freely. It also facilitates providing complex multi-step instructions, while reducing the complexity of computer circuits.</dd></dl>
